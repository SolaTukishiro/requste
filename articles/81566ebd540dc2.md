## はじめに

Laravelで個人開発を進めていると、  
Bladeテンプレートの中でこんな記述をよく見かけます。

```php
<x-app-layout>
<x-nav-link>
@include('...')
```

最初は  
「なんとなく使ってはいるけど、正直よくわかっていない」  
状態でした。

特に

- <x-...> はどこを見に行っているのか？
- layouts と components の違いは？
- @include() との使い分けは？

がごちゃごちゃになっていたので、  
初学者向けに**考え方ベース**で整理します。

## <x-...> の正体は Bladeコンポーネント

結論から言うと、
```php
<x-foo-bar>
```

と書いたとき、Bladeは **決められた順番でファイルを探しているだけ** です。  

### Bladeの探索ルール（超重要）

```php
<x-foo-bar>
```
を見つけると、Bladeは次の順で探します。  

1. クラス付きコンポーネント
```md
 app/View/Components/FooBar.php
```
2. viewコンポーネント
```md
resources/views/components/foo-bar.blade.php
```
どちらかが見つかれば、それを表示します。

### <x-app-layout> が layouts を表示する理由
最初に混乱したのがここでした。

**「x-app-layout なのに、なんで layouts フォルダを見るの？」**

理由はシンプルで、  
クラス側で表示する view を指定しているからです。

```php
// app/View/Components/AppLayout.php
public function render()
{
    return view('layouts.app');
}
```
つまり
```md
<x-app-layout>
 → AppLayout クラス
 → layouts/app.blade.php を表示
```
という流れになっています。  
Bladeが「layouts だから特別扱い」しているわけではありません。

### フォルダ名は Blade にとって重要ではない

ここが一番の気づきでした。

- layouts
- components
- partials
- includes

これらのフォルダ名は  
→**人間が管理やすくするための名前であって、 Bladeの判断基準ではない**

Bladeにとって重要なのは、  
- クラスがあるか？
- 決められたパスにあるか？
- 名前が一致しているか？

だけです。

### じゃあ @include() は何？
@include() はとても素直な仕組みです。
```md
@include('client.requests.index')
```
これは、
```php
return view('client.requests.index');
```

の **Blade版** だと考えると理解しやすいです。

### 役割の違いを整理すると
| 使い方   | 役割 |
|--| - |
| view() | Controllerからページ全体を返す|
| @include() | Blade内で部品を貼り付ける |
|<x-...>|再利用・状態を持つ部品|

### 使い分けの指針（初学者向け）
迷ったら、こう考えると楽でした。
```md
状態を持たせたい / 再利用したい  
→ components + <x-...>

HTMLの塊を分けたいだけ  
→ include + 自由なフォルダ
```
無理に全部コンポーネントにする必要はありません。

## まとめ
今回理解できたポイントは以下です。

- <x-...> は Blade が自動で探している
- クラスがあればクラス付きコンポーネント
- なければ resources/views/components を見る
- フォルダ名自体は重要ではない
- @include() は Blade版の view() みたいなもの

これが分かってから、  
Laravelの「魔法っぽさ」が一気に減りました。
